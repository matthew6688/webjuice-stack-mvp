# ProfitsLocal Website Intake (M1)

Single skill that owns *all* business intake for the ProfitsLocal pipeline. V3 主路径 = Discord listener (operator 在 #website-tasks forum 发命令)。

**Note**: V3 production 99% 操作走 Discord listener (intent-router cascade · 不调本 skill)。本 skill 仅在 Matthew 跟 Hermes 自然对话时被加载。两路径殊途同归 · 都最终调 npm run pl:* 同一组 CLI。

---

## 4 entries · pick by input shape

| Input pattern | CLI |
|---|---|
| Natural-language `"find {niche} in {city}"` (no quotes) | `npm run pl:pipeline-batch-start -- --niche {X} --city {Y} --count 10` |
| Multiple quoted queries `"X" "Y"` | `npm run pl:places-search-intake -- --query "{X}" --query "{Y}"` |
| Single business · name + phone (+/- Maps URL) | `npm run pl:single-enrich -- --business-name "..." --phone {...} --city {...}` |
| Image attachment (card / shopfront / screenshot) | `npm run pl:ingest-image -- --image {path} --business-name {...} --niche {...} --city {...}` |

## Behaviour rules

- `niche` + `city` are mandatory · extract them from the user message. If you cannot extract both, ask the user before calling any CLI.
- City extraction is LLM-driven (not regex). Don't bail just because the city isn't in a hardcoded list — brisbane / sydney / melbourne / cairns / newcastle / perth / adelaide / hobart / canberra / darwin / gold coast / townsville and any other AU city is fair game.
- For a single business with phone or Maps URL, prefer `pl:single-enrich`.
- **For images (D40 update)**: the listener will OCR the image first AND automatically run multi-angle Places enrich. Most cases the OCR + enrich extracts business_name + city + place_id without operator补字段. Only ask operator if extraction COMPLETELY fails (no phone + no name extracted).
- Dedup (8-key) happens inside the entity store · callers don't manage it.

---

## Gotchas

- `pl:pipeline-batch-start` chains to `pl:scrape-docker`. It needs lat/lng but geocodes internally; you just supply city.
- `pl:single-enrich` currently requires `business-name` + at least one of `phone` / Maps URL. Pure-phone / pure-URL is supported via D40 image-enrich path (用 enrichFromOCR module 直接调 multi-angle search · 但 single-enrich CLI 本身仍要求 name + phone).
- `pl:ingest-image` requires the image file to exist on disk OR `--dry-run`.
- Image OCR 缺关键字段时 · D40 自动 fallback 到 Places multi-angle 搜索 · 通常救活 · 仅完全失败时才进 human gate。

---

## Output contract

After a successful intake call:
- An entity lands in `data/leads/entities/<entityKey>.json`.
- A `master.md` skeleton is auto-built within 30s (per-entity).
- The entity enters the SOP-2 audit queue (rescored on the next 4h cron · or manually trigger `npm run leads:run-pipeline -- --entity-key X`).
- M2 audit auto-chain · 5 stage Discord 通知 (D38) · contact extraction (D37) · qualification check (D39 实装后)。
- 仅 entity qualified (Score ≥ 60) 才 chain pl:build-from-reference + pl:publish-demo (D39 · 在 backlog · 尚未实装时所有 grade ABC 都 auto-build)。

## Discord 进度通知 (V3 D38+D40)

每个 intake 任务在 Discord 自动发以下消息:

1. **#website-tasks · 任务创建** (listener · 1 条 ack)

3. **#website-tasks · 任务完成** (dispatcher · 列商家名 不显 place_id)
